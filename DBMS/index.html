UNIT I: Introduction & ER Model

Foundation

This unit forms the foundation of database management. It covers the transition from traditional file systems to robust database systems, data models, architectural independence, and the visual representation of data using the Entity-Relationship (ER) model.

2-Mark Questions

1. Define a Database Management System (DBMS).

A DBMS is a software system that enables users to define, create, maintain, and control access to a database.
Example: MySQL or Oracle managing student records in a university portal.

2. What is Data Independence?

It is the ability to modify the schema at one level of the database architecture without altering the schema at the next higher level.
Example: Changing the hard drive storage (Physical level) without modifying the application code (Logical level).

3. Differentiate between an Entity and an Attribute.

An entity is a real-world object with independent existence, while an attribute is a property that describes that entity.
Example: In a college database, "Student" is an entity, and "Roll Number" is an attribute.

10-Mark Questions

1. Explain the differences between a Database System and a File System. Highlight the advantages of DBMS.

  • Definition: A file system manages data in disparate files, whereas a DBMS integrates data in a centralized repository.
  • Detailed Explanation: File systems suffer from data redundancy (duplicate data), data inconsistency, difficulty in access, and lack of security. DBMS solves this by offering centralized control, data sharing, backup, and concurrency control.
  • Diagram Explanation: Draw a two-part diagram. On the left, show multiple users accessing different independent files (File System). On the right, show multiple users interacting with a central "DBMS" circle, which then points to the "Database" storage.
  • Real-time Use Case: A banking system. If an address changes, a DBMS updates it once centrally, whereas a file system might require updating the "Loans" file and the "Savings" file separately.
  • Conclusion: Transitioning to a DBMS is mandatory for enterprise applications due to data integrity and security benefits.

2. Describe the Three-tier Schema Architecture for Data Independence.

  • Definition: An architecture that separates the user applications from the physical database, divided into internal, conceptual, and external levels.
  • Detailed Explanation:
    - Internal Level: Deals with physical storage.
    - Conceptual Level: Describes what data is stored and relationships (Community view).
    - External Level: Describes the part of the database relevant to specific users (User views).
  • Diagram Explanation: Draw three horizontal layers. Top: Multiple User Views. Middle: Conceptual Schema. Bottom: Internal Schema pointing to Physical Storage. Connect them with mapping arrows.
  • Real-time Use Case: A university portal. Students only see their grades (External), admins see all student data (Conceptual), and DBAs manage server space (Internal).
  • Conclusion: This architecture ensures that changes in storage hardware do not break front-end user applications.

3. What is an ER Model? Explain the representation of entities, attributes, and relationships with an ER Diagram.

  • Definition: The ER Model is a high-level conceptual data model that represents the logical structure of a database visually.
  • Detailed Explanation:
    - Entity: Represented by a rectangle.
    - Attribute: Represented by an oval (Key attributes are underlined).
    - Relationship: Represented by a diamond connecting entities.
  • Diagram Explanation: Draw a rectangle "Customer" connected by a diamond "Deposits" to a rectangle "Account". Draw ovals from "Customer" for "Name" and "Cust_ID" (underlined).
  • Real-time Use Case: E-commerce database designing where 'Users' place 'Orders' containing 'Products'.
  • Conclusion: ER diagrams are the universal blueprint used by architects before writing any SQL code.

4. Discuss Specialization, Generalization, and Inheritance in ER Models.

  • Definition: These are extended ER features used to model complex hierarchical data.
  • Detailed Explanation:
    - Specialization: Top-down approach. Splitting a higher-level entity into lower-level entities.
    - Generalization: Bottom-up approach. Combining lower-level entities into a higher-level entity based on common attributes.
    - Inheritance: Lower-level entities inherit attributes from their superclass.
  • Diagram Explanation: Draw a triangle (labeled "ISA") pointing up to a superclass (e.g., "Employee") and down to subclasses (e.g., "Full-time", "Part-time").
  • Real-time Use Case: An HR Management system categorizing different types of employees who share basic details but have specific benefits.
  • Conclusion: These concepts reduce attribute redundancy and apply object-oriented principles to database design.

5. Detail the Database System Structure and Environment.

  • Definition: The systematic breakdown of a database environment into its functional components and users.
  • Detailed Explanation: The system consists of query processors (DML compiler, DDL interpreter) and storage managers (buffer manager, transaction manager). It operates in environments ranging from centralized to client-server architectures.
  • Diagram Explanation: Draw a block diagram showing "Users" at the top, passing queries to the "Query Processor", which communicates with the "Storage Manager", ending at the "Disk Storage" at the bottom.
  • Real-time Use Case: A corporate data center where multiple clients (front-end apps) connect to a robust Oracle server (back-end).
  • Conclusion: Understanding the system structure is vital for database administrators to optimize performance and troubleshoot bottlenecks.

UNIT II: Relational Model & Basic SQL

Logical Design

This unit transitions from conceptual design to logical structures using the Relational Model. It covers constraints, relational algebra, and the foundational elements of Basic SQL (DDL and DML).

2-Mark Questions

1. Define Tuple and Domain in a Relational Model.

A tuple is a single row or record in a table. A domain is the set of all permitted values for a given attribute.
Example: In an Age column, the domain might be integers from 1 to 120.

2. What is the importance of NULL values?

NULL represents a missing, unknown, or inapplicable value, not zero or a blank space.
Example: A "Middle Name" column might be NULL if a person doesn't have one.

3. List any three DML operations.

INSERT (adds data), UPDATE (modifies data), and DELETE (removes data).

10-Mark Questions

1. Explain the various constraints in the Relational Model (Domain, Key, Integrity).

  • Definition: Constraints are rules enforced on data to ensure its accuracy, validity, and reliability.
  • Detailed Explanation:
    - Domain Constraint: Ensures data in a column meets the data type criteria.
    - Key Constraint: Ensures every row is uniquely identifiable (Primary Key).
    - Integrity Constraint (Referential): Ensures relationships between tables are valid via Foreign Keys.
  • Diagram Explanation: Show two tables: "Department" and "Employee". Draw an arrow from "Dept_ID" in the Employee table to "Dept_ID" in the Department table to visualize Referential Integrity.
  • Real-time Use Case: A shopping cart system. You cannot add an item to an order if that item ID does not exist in the master Inventory table.
  • Conclusion: Constraints are the primary defense against data corruption in relational databases.

2. Discuss Relational Algebra and explain its fundamental operations.

  • Definition: A procedural query language consisting of a set of operations that take one or two relations as input and produce a new relation.
  • Detailed Explanation:
    - Select (σ): Filters rows based on a condition.
    - Project (π): Filters columns.
    - Union (∪): Combines tuples from two relations.
    - Set Difference (-): Tuples in relation A but not in B.
    - Cartesian Product (X): Combines every tuple of A with every tuple of B.
  • Diagram Explanation: Use Venn diagrams to show Union (both circles filled) and Set Difference (only the left circle filled minus the intersection).
  • Real-time Use Case: An airline app filtering available flights (Select) and showing only flight numbers and prices (Project).
  • Conclusion: Relational algebra forms the theoretical foundation upon which SQL execution engines are built.

3. Explain DDL commands in SQL with syntax and examples.

  • Definition: Data Definition Language (DDL) defines the structure or schema of the database.
  • Detailed Explanation:
    - CREATE: Makes a new table. CREATE TABLE Student (ID int, Name varchar(50));
    - ALTER: Modifies structure. ALTER TABLE Student ADD Age int;
    - DROP: Deletes the table completely. DROP TABLE Student;
    - TRUNCATE: Empties the table but keeps structure.
  • Diagram Explanation: N/A for pure SQL commands, but you can draw a visual representation of an empty table structure being created.
  • Real-time Use Case: A software release where the database schema must be upgraded by adding a new "Discount_Code" column to the "Payments" table using ALTER.
  • Conclusion: DDL commands are auto-committed and must be used with caution as they alter the system blueprint.

4. Describe DML operations with examples.

  • Definition: Data Manipulation Language (DML) manages data within existing schema objects.
  • Detailed Explanation:
    - INSERT: INSERT INTO Student VALUES (1, 'John');
    - UPDATE: UPDATE Student SET Name = 'Doe' WHERE ID = 1;
    - DELETE: DELETE FROM Student WHERE ID = 1;
  • Diagram Explanation: Show a tabular grid. Highlight a row being added (INSERT), a cell changing color (UPDATE), and a row being crossed out (DELETE).
  • Real-time Use Case: A social media app where a user posts a comment (INSERT), edits it (UPDATE), or removes it (DELETE).
  • Conclusion: DML operations are the most frequently executed commands in any active software application.

5. Compare Relational Algebra and Relational Calculus.

  • Definition: Relational algebra is procedural (how to get data), while relational calculus is non-procedural/declarative (what data to get).
  • Detailed Explanation: Algebra uses mathematical symbols to describe the exact steps (Select, Project, Join). Calculus uses logic formulations (Tuple Relational Calculus and Domain Relational Calculus) describing the desired result without specifying the steps.
  • Diagram Explanation: Create a comparison table outlining Procedural vs Non-Procedural, Sequence of operations, and examples of syntax.
  • Real-time Use Case: SQL is heavily inspired by Relational Calculus (stating what you want), while the internal database query optimizer uses Relational Algebra to figure out how to get it.
  • Conclusion: A strong grasp of both is essential for understanding query optimization engines.

UNIT III: SQL Querying

Practical SQL

This unit focuses on practical database interaction. It covers advanced SQL, including functions, complex joins, nested queries, aggregation, and the implementation of views.

2-Mark Questions

1. What is the purpose of the GROUP BY clause?

It is used in collaboration with aggregate functions to group rows that have the same values into summary rows.
Example: SELECT Dept, SUM(Salary) FROM Employee GROUP BY Dept;

2. Define a View in SQL.

A view is a virtual table based on the result-set of an SQL statement. It does not store data physically.

3. What is a Subquery?

A query nested inside another query, used to restrict or enhance the data to be queried by the main query.

10-Mark Questions

1. Explain the different types of Joins in SQL with examples.

  • Definition: A JOIN clause is used to combine rows from two or more tables based on a related column.
  • Detailed Explanation:
    - INNER JOIN: Returns records with matching values in both tables.
    - LEFT JOIN: Returns all records from the left table, and matched records from the right.
    - RIGHT JOIN: Returns all records from the right table, and matched records from the left.
    - FULL OUTER JOIN: Returns all records when there is a match in either table.
  • Diagram Explanation: Draw Venn diagrams for each. Inner Join = intersection. Left Join = left circle fully shaded + intersection, etc.
  • Real-time Use Case: An E-commerce dashboard joining the "Users" table and "Orders" table to show a customer's name next to their purchase history.
  • Conclusion: Joins are fundamental for retrieving meaningful information from normalized databases.

2. Discuss Aggregate Functions and Grouping in SQL.

  • Definition: Aggregate functions perform calculations on multiple rows of a single column to return a single value.
  • Detailed Explanation: Key functions include COUNT(), SUM(), AVG(), MAX(), and MIN(). These are heavily used with the GROUP BY clause. The HAVING clause is used to filter grouped records (since WHERE cannot be used with aggregates).
  • Diagram Explanation: Show a table of employees with different departments and salaries. Draw an arrow pointing to a smaller summary table showing just Departments and their Total Salaries.
  • Real-time Use Case: A corporate financial report calculating the total payroll cost (SUM) for each department (GROUP BY).
  • Conclusion: Mastering aggregation is critical for data analysis and generating business intelligence reports.

3. What are Nested Queries? Explain with syntax.

  • Definition: A nested query (subquery) is a query within another SQL query, embedded within the WHERE, HAVING, or FROM clause.
  • Detailed Explanation: The inner query executes first, and its result is passed to the outer query.
    Syntax Example: SELECT Name FROM Employee WHERE Salary > (SELECT AVG(Salary) FROM Employee);
  • Diagram Explanation: Draw an outer box (Main Query) containing an inner box (Subquery). Show an arrow indicating data flowing from the inner box to the outer box.
  • Real-time Use Case: Finding all students who scored higher than the class average. The average must be calculated first (inner query) before comparing (outer query).
  • Conclusion: Nested queries solve complex multi-step logical problems within a single SQL statement.

4. Explain Views in SQL. Differentiate between updatable and non-updatable views.

  • Definition: A view is a stored query accessible as a virtual table.
  • Detailed Explanation:
    - Updatable View: Allows DML operations (Insert/Update/Delete) that reflect directly on the base table. Usually, these views draw from a single table without aggregates.
    - Non-updatable View: Read-only views. They usually contain Joins, GROUP BY, or Aggregate functions, meaning the DBMS cannot map a change back to a specific base row.
  • Diagram Explanation: Show a "User" looking through a "Window (View)" to see a subset of the "Base Tables".
  • Real-time Use Case: Creating a "Public_Employee_Directory" view that only shows Names and Emails, hiding the base table's Salary and SSN columns from normal staff.
  • Conclusion: Views are an excellent tool for enhancing security and simplifying complex query access for end-users.

5. Describe SQL Functions (Date/Time, Numeric, String conversion).

  • Definition: Built-in SQL utilities used to manipulate data items and return transformed values.
  • Detailed Explanation:
    - String: UPPER(), LENGTH(), SUBSTRING().
    - Numeric: ROUND(), ABS(), POWER().
    - Date/Time: SYSDATE(), DATEDIFF(), MONTH().
  • Diagram Explanation: Create a 3-column table categorizing the function types, with a brief example next to each.
  • Real-time Use Case: Calculating the exact age of a user in an application by using Date functions to find the difference between SYSDATE() and their stored Date of Birth.
  • Conclusion: Using database-level functions is often faster and more efficient than manipulating data in the application backend layer.

UNIT IV: Schema Refinement

Normalization

This unit tackles database design flaws. It covers the mathematical process of Normalization to eliminate data redundancy and insertion/deletion anomalies using functional dependencies.

2-Mark Questions

1. What is the purpose of Normalization?

To refine a database schema to minimize redundancy and prevent insertion, deletion, and update anomalies.

2. Define Functional Dependency.

A relationship where one attribute uniquely determines another. If X determines Y (X -> Y), X is the determinant.

3. What is a Surrogate Key?

An artificially generated key (like an auto-incrementing integer) used when natural columns do not form a good primary key.

10-Mark Questions

1. Explain 1NF, 2NF, and 3NF with examples.

  • Definition: The sequential rules of normalization to organize data efficiently.
  • Detailed Explanation:
    - 1NF: Ensures attributes are atomic (no multi-valued attributes or repeating groups).
    - 2NF: Must be in 1NF. Removes partial dependencies (non-prime attributes must depend on the entire primary key).
    - 3NF: Must be in 2NF. Removes transitive dependencies (non-prime attributes must not depend on other non-prime attributes).
  • Diagram Explanation: Draw a single massive table breaking down into three separate, cleaner tables as it passes through the normal forms.
  • Real-time Use Case: A student registration system. Moving from a messy spreadsheet where a student has "Multiple Phone Numbers" in one cell (violates 1NF) to separate properly linked tables.
  • Conclusion: Achieving 3NF is considered the standard requirement for most business applications.

2. Discuss Boyce-Codd Normal Form (BCNF). How is it different from 3NF?

  • Definition: BCNF is a stricter version of 3NF. For every functional dependency X -> Y, X must be a super key.
  • Detailed Explanation: A table can be in 3NF but fail BCNF if a prime attribute depends on a non-prime attribute (usually happens when there are overlapping candidate keys).
  • Diagram Explanation: Show a table schema: Student, Course, Instructor. Instructor determines Course. If an Instructor only teaches one Course, Instructor -> Course. Since Instructor is not a super key, it violates BCNF despite being in 3NF.
  • Real-time Use Case: University scheduling where multiple professors teach overlapping subjects.
  • Conclusion: BCNF ensures absolute elimination of redundancy caused by overlapping candidate keys.

3. What is a Lossless Join and Dependency Preserving Decomposition?

  • Definition: Properties of a good decomposition when breaking a large table into smaller normalized tables.
  • Detailed Explanation:
    - Lossless Join: When decomposed tables are natural-joined, they must yield the exact original table without creating "spurious" (fake) tuples.
    - Dependency Preserving: All original functional dependencies must be checkable in the new tables without needing a join.
  • Diagram Explanation: Draw Table R splitting into R1 and R2. Show an equation: R1 JOIN R2 = R.
  • Real-time Use Case: Refactoring a legacy corporate database. You must ensure splitting tables doesn't lose historical connections or violate existing business rules.
  • Conclusion: Decompositions must always be lossless; dependency preservation is highly desired but sometimes sacrificed for BCNF.

4. Explain Multi-Valued Dependency (MVD) and Fourth Normal Form (4NF).

  • Definition: 4NF deals with multi-valued dependencies where one attribute independently multi-determines two other attributes.
  • Detailed Explanation: If a teacher teaches multiple Subjects AND participates in multiple Hobbies, storing them in one table creates massive redundancy (Cartesian product of subjects and hobbies). 4NF requires splitting this into two tables: (Teacher, Subject) and (Teacher, Hobby).
  • Diagram Explanation: Show a badly formed table with repeating combinations of subjects and hobbies, then arrows splitting it into two distinct, smaller tables.
  • Real-time Use Case: A restaurant database where a restaurant offers multiple delivery areas AND multiple distinct cuisines.
  • Conclusion: 4NF is essential when an entity has multiple, independent 1-to-Many relationships.

5. Briefly describe the concept of Fifth Normal Form (5NF).

  • Definition: Also known as Project-Join Normal Form (PJNF). It deals with cases where information can be reconstructed from smaller pieces that can be joined together.
  • Detailed Explanation: A table is in 5NF if it is in 4NF and cannot be decomposed into smaller tables without losing data upon rejoining (Join Dependency). It usually applies to complex ternary (3-way) relationships.
  • Diagram Explanation: Draw a 3-column table (Supplier, Part, Project) and show it decomposing into three 2-column tables to avoid redundancy.
  • Real-time Use Case: Supply chain management where specific suppliers only provide specific parts to specific projects.
  • Conclusion: 5NF is rarely implemented practically but represents the theoretical pinnacle of database normalization.

UNIT V: Transactions & Indexing

Advanced Concepts

This final unit covers the dynamic aspects of databases: how simultaneous transactions are managed safely (Concurrency/ACID properties), how systems recover from crashes, and how to retrieve data quickly using Indexing.

2-Mark Questions

1. What are the ACID properties?

Atomicity, Consistency, Isolation, and Durability. They guarantee transaction reliability.

2. Define a Deadlock.

A situation where two or more transactions are waiting indefinitely for each other to release locks.

3. What is the purpose of B+ Trees in databases?

B+ trees are used for dynamic multilevel indexing to allow fast, balanced search, insertion, and deletion of records.

10-Mark Questions

1. Explain the ACID properties of a Transaction in detail.

  • Definition: A transaction is a logical unit of work. ACID ensures its reliable execution.
  • Detailed Explanation:
    - Atomicity: "All or nothing." The transaction completes entirely or leaves the database unchanged.
    - Consistency: The database transitions from one valid state to another.
    - Isolation: Concurrent transactions execute as if they were serial.
    - Durability: Committed changes survive system crashes.
  • Diagram Explanation: Create a 4-quadrant box with A, C, I, D and a one-word keyword in each (e.g., A=Entirety, C=Validity).
  • Real-time Use Case: Transferring $100 from Account A to B. Atomicity ensures A isn't debited without B being credited if the server crashes mid-transfer.
  • Conclusion: ACID is the absolute core mechanism that prevents financial and data catastrophes in software.

2. Describe Transaction States with a state transition diagram.

  • Definition: The life cycle of a transaction from initiation to completion or failure.
  • Detailed Explanation: The states are: Active, Partially Committed (after last statement execution), Committed (successful save), Failed (hardware/logic error), and Aborted (rolled back).
  • Diagram Explanation: Draw circles for each state. Start at "Active". Branch left to "Failed" -> "Aborted". Branch right to "Partially Committed" -> "Committed".
  • Real-time Use Case: An online ticket booking. Active while you select seats. Failed if your card declines. Committed when the confirmation email sends.
  • Conclusion: The state machine model allows DBMS recovery managers to know exactly when to roll back or permanently write data.

3. Discuss Concurrency Control protocols (Lock-based and Timestamp-based).

  • Definition: Mechanisms to ensure that multiple transactions can occur simultaneously without violating Isolation.
  • Detailed Explanation:
    - Lock-Based: Uses Shared (Read) and Exclusive (Write) locks. Two-Phase Locking (2PL) dictates a growing phase (acquiring locks) and a shrinking phase (releasing locks) to ensure serializability.
    - Timestamp-Based: Assigns a unique system time to every transaction. Older transactions get priority over younger ones to prevent conflicts without using locks.
  • Diagram Explanation: Draw a timeline for 2PL showing a rising slope (Growing Phase), a peak (Locked), and a falling slope (Shrinking Phase).
  • Real-time Use Case: Flash sales on E-commerce sites. Thousands of users trying to buy the last iPhone simultaneously require strict locking to prevent overselling.
  • Conclusion: Choosing the right protocol balances data safety with system throughput (speed).

4. Explain the concept of Database Recovery Algorithms.

  • Definition: Techniques used to restore the database to a consistent state after a system failure.
  • Detailed Explanation: Involves Log-based recovery.
    - Deferred Update: Doesn't physically update the database until the transaction commits (only writes to log). No UNDO required, only REDO.
    - Immediate Update: Updates physical database as the transaction progresses. Requires both UNDO (for uncommitted) and REDO (for committed).
  • Diagram Explanation: Show a timeline with a "System Crash" point. Show transactions that finished before the crash being REDOne, and those active during the crash being UNDOne.
  • Real-time Use Case: A power outage strikes a server room. Upon reboot, the DBMS reads the logs and instantly rolls back incomplete operations.
  • Conclusion: Recovery algorithms ensure Durability (the 'D' in ACID).

5. Introduce Indexing Techniques: Detail B+ Trees and Hash-Based Indexing.

  • Definition: Indexing creates a data structure that improves the speed of data retrieval operations.
  • Detailed Explanation:
    - B+ Trees: A balanced tree structure where all data pointers are stored at the leaf nodes, which are linked sequentially. Excellent for range queries.
    - Hash-Based Indexing: Uses a hash function to compute the exact bucket location of a record based on its key. Extremely fast for exact-match queries.
  • Diagram Explanation: For B+ Trees, draw a root node pointing to internal nodes, pointing to a linked list of leaf nodes.
  • Real-time Use Case: Searching for an employee ID 105 (Hash is faster). Searching for employees hired between 2020 and 2023 (B+ Tree is faster).
  • Conclusion: Proper indexing is the most effective way a database administrator can tune query performance in large-scale databases.
